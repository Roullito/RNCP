## Module 5 – Méthodologie du questionnaire professionnel

### 5.1. Rôle et format du questionnaire professionnel

Le **questionnaire professionnel** est la **première épreuve** que tu passes le jour J.

- Durée : **30 minutes**.
- Format (pour le nouveau référentiel) :
  - **2 questions fermées (QCM) en français**
  - **2 questions ouvertes en anglais**
- Support : une **documentation technique jointe** (PDF, extrait de doc, fiche mémoire…).
- Objectif du jury :
  - Vérifier ta **capacité à lire et comprendre une documentation technique**.
  - Vérifier ta **maîtrise des concepts de base du métier** (en lien avec le REAC).
  - Vérifier ta capacité à **t’exprimer simplement en anglais technique** (questions ouvertes).

> ⚠️ Remarque :  
> L’**intro du projet en anglais** a disparu.  
> L’anglais est désormais évalué **via ces questions ouvertes** du questionnaire professionnel.

---

### 5.2. Ce qui est évalué derrière le questionnaire

Le questionnaire pro teste notamment :

1. **Tes bases techniques** :
   - Environnement de développement : OS, éditeur, Git, Node.js, etc.
   - Concepts web : client / serveur, HTTP, API, etc.
   - Outils du développeur : GitHub, frameworks, etc.

2. **Ta capacité à lire une documentation** :
   - Savoir trouver une information précise.
   - Comprendre une définition, un schéma, un exemple.

3. **Ta capacité à restituer en anglais simple** :
   - Expliquer une notion (ex : *What features does GitHub provide?*).
   - Reformuler une définition ou un concept (ex : *What is major/central concept of Node.js?*).

---

### 5.3. Méthode pour les questions fermées (QCM en français)

Les QCM sont toujours formulés avec :  
> “En vous référant à la documentation technique jointe…”

Exemples donnés :

- “En vous référant à la documentation technique, quel outil n’est pas un besoin dans un environnement de développement ? (Git / Linux / NodeJS…)”
- “En vous référant à la documentation technique, quelle instruction git permet d’ajouter de nouveaux fichiers ? (mkdir / git-push / git-add…)”

**Stratégie :**

1. **Ne pas répondre “au feeling” d’abord.**  
   → Cherche d’abord dans la **documentation fournie** (titres, tableaux, listes).

2. Utiliser une **lecture ciblée** :
   - Repérer les **mots-clés** de la question (ex : “environnement de développement”, “Git”, “ajouter de nouveaux fichiers”).
   - Scanner la doc pour trouver la section correspondante.

3. Vérifier :
   - Souvent, la bonne réponse est **écrite mot pour mot** dans la doc.
   - Si tu hésites, demande-toi :
     - “Est-ce que cet élément fait vraiment partie de l’environnement de dev décrit dans la doc ?”
     - “Est-ce que la commande mentionnée existe vraiment dans Git / Linux ?”

4. Répondre proprement :
   - Cocher **une seule réponse**, sauf si le QCM précise plusieurs réponses possibles.

---

### 5.4. Méthode pour les questions ouvertes en anglais

Exemples donnés :

- **What features does Github provide ?**
- **What is major/central concept of Node.js ?**

Objectif :  
> Montrer que tu peux **lire la doc technique** et **répondre en anglais simple, correct, et compréhensible**.

**Stratégie :**

1. **Repérer la section correspondante dans la documentation** :
   - Pour GitHub : chercher des mots comme *“GitHub features”, “repositories”, “issue tracking”, “collaboration”*…
   - Pour Node.js : chercher *“event loop”, “non-blocking I/O”, “asynchronous”*, etc.

2. **Extraire 2–3 idées clés** :
   - Pas besoin de recopier tout un paragraphe.
   - Tu cherches **les idées principales**, pas tous les détails.

3. **Répondre en anglais simple** :
   - Phrases courtes.
   - Sujet + verbe + complément.
   - Pas de phrases hyper compliquées.

4. **Exemple de structure de réponse** (générique) :

   - *“GitHub provides online code hosting, version control, and collaboration tools. It allows developers to share projects, review code, and manage issues.”*  
   - *“The central concept of Node.js is non-blocking, event-driven I/O. It lets JavaScript handle many connections at the same time with a single thread.”*

> Le but n’est pas d’être “fluent”, mais de répondre **clairement**, **sans fautes bloquantes**, et en lien direct avec la doc.

### 5.5. Lien avec le REAC DWWM

Le questionnaire professionnel est en lien avec plusieurs compétences du **REAC DWWM** :

- **CP1** – Installer et configurer l’environnement de travail  
  → Questions sur les outils (Git, Node, OS, IDE, etc.)
- **Compétences transversales** :
  - Lire et exploiter une **documentation technique**.
  - Communiquer en anglais dans un contexte professionnel (niveau simple).
- **Connaissances web / back-end** :
  - Concepts client/serveur, API, Node.js, etc.

> Le jury ne teste pas un framework précis, mais ta capacité à **comprendre un environnement de développeur web moderne**.

---

### 5.6. Plan de révision pour le questionnaire pro

Pour te préparer efficacement :

1. **Revoir l’environnement de développement** :
   - Git : commandes de base (`clone`, `add`, `commit`, `push`, `pull`, `status`…)
   - GitHub : repos, branches, issues, pull requests.
   - Node.js : rôle, environnement d’exécution JS côté serveur, event loop.
   - Outils classiques : OS, IDE, terminal, navigateur, etc.

2. **T’entraîner à lire une doc technique** :
   - Prendre une page de doc (Git, GitHub, Node, MDN…) et t’entraîner à :
     - Chercher une info précise à partir d’une question.
     - Reformuler en 2–3 phrases.

3. **T’entraîner aux réponses en anglais simple** :
   - T’écrire des mini fiches :
     - “What is Git?”  
     - “What is GitHub used for?”  
     - “What is an API?”  
     - “What is Node.js?”  
   - Pour chaque question, rédiger **2–3 phrases max**, lisibles et mémorisables.

4. **Simuler un questionnaire** :
   - Te mettre 30 min.
   - Prendre 2 QCM FR + 2 questions ouvertes EN (tu peux t’en inventer).
   - T’entraîner à gérer le temps.
